# orchestrator/lib/execute.sh — core invocation
run_claude_step() {
  local step_file="$1" task_file="$2" context_file="$3"
  local log_file="$4" summary_file="$5"
  local timeout_seconds="$6" repo_root="$7"

  local permission_mode
  local -a claude_args=()
  permission_mode=$(sdlc_lookup_kv STEP_PERMISSION_MODES \
                   "$step_name" "$CLAUDE_PERMISSION_MODE")

  if [[ -n "$CLAUDE_EXTRA_ARGS" ]]; then
    # shellcheck disable=SC2206
    local -a extra_args=($CLAUDE_EXTRA_ARGS)
    claude_args+=("${extra_args[@]}")
  fi

  # ... prompt assembly unchanged ...

  sdlc_log "INFO" "Model: $CLAUDE_MODEL | Effort: $CLAUDE_EFFORT | Permission mode: $permission_mode"

  set +e
  (
    cd "$repo_root"
    printf '%s' "$full_prompt" | sdlc_run_with_timeout "$timeout_seconds" \
      claude \
        --print \
        --model "$CLAUDE_MODEL" \
        --effort "$CLAUDE_EFFORT" \
        --permission-mode "$permission_mode" \
        --output-format "$CLAUDE_OUTPUT_FORMAT" \
        ${claude_args[@]+"${claude_args[@]}"} \
        2>&1 | tee "$log_file"
    # claude is PIPESTATUS[1]; PIPESTATUS[0] is always printf's success
    exit "${PIPESTATUS[1]}"
  )
  local exit_code=$?
  set -e

  # --print streams the last assistant message to stdout, which we tee'd.
  if [[ -s "$log_file" ]]; then
    cp "$log_file" "$summary_file"
  fi

  return "$exit_code"
}

Behavioral impact

  • Hard prerequisite changes: the pipeline now requires claude on PATH. Any target repo that had codex but not claude will now fail at the prereq gate.
  • Default sandboxing model changes meaning: --full-auto (Codex) is replaced by --permission-mode acceptEdits (Claude). The semantics differ — Claude's permission model is about what tool calls require prompts, not a sandbox directory; reviewers should confirm acceptEdits is the right default for every step (Step 9, semantic diff, is read-mostly — consider tightening).
  • Summary file contents change: previously --output-last-message wrote only the final assistant message; now cp $log_file $summary_file writes everything tee'd, which in text mode happens to also be the last assistant message, but will diverge if someone enables a verbose output format. Downstream update_context consumes this file.
  • Working directory behavior: the subshell cd $repo_root replaces codex exec -C. Because the subshell still executes after set +e returns from the pipeline, and the subshell's own exit is what the parent reads, the retry loop still sees the claude exit code correctly.

Reviewer scrutiny

  • Does --print + --output-format text really emit only the last assistant message? If it emits tool-call traces or progress text, $summary_file becomes noisy and update_context will propagate that noise to the next step's prompt. No assertion in tests pins this down.
  • CLAUDE_EXTRA_ARGS word-splitting uses bare $CLAUDE_EXTRA_ARGS inside (...). Values containing spaces inside quotes (e.g., --system-prompt "hello world") will be split. Document as a known limitation or move to an array-valued override.
  • Permission-mode validation. sdlc_lookup_kv returns whatever string the overrides file supplies. A typo like acceptEdit will be passed straight through to claude, which will fail late. Consider a whitelist.

Objective

Why this change exists

Running it after Step 10's cleanup means the reviewer sees the remediated code, not intermediate state.

Justification audit

  • Strong Mandatory ## Actions section with fix / defer / reject + rationale. Without this, /ultrareview becomes advisory; with it, the pipeline cannot silently drop findings.
  • Strong STEP_TIMEOUTS entry of 3600s and STEP_RETRY_COUNTS of 2. The 60-minute timeout matches the Step 10 precedent; 2 retries prevents a flaky network from burning the run.
  • Probable Step 11 triggers a full-suite re-run (delegating to Step 6) after applied fixes. Correct — any accepted fix can regress tests — but step text says "re-run the test suite (Step 6)" without a concrete command or gate. Relies on the acting model to remember.

# Step 11 — Ultra Review

**Mode:** Automated
**Objective:** Run a dedicated review session that reads through the
branch changes and flags bugs and design issues a careful reviewer would catch.

## Inputs
- Current feature branch diff against `main`
- `.sdlc/reports/semantic_diff_report_<ticket-id>.html` from Step 9
- `.sdlc/artifacts/semantic-review-actions.md` from Step 10
- Canonical output file: `.sdlc/artifacts/ultra-review.md`

## Procedure

1. Invoke Claude Code's `/ultrareview` slash command:
   ```
   claude -p "/ultrareview" --permission-mode acceptEdits
   ```

2. Capture the review output verbatim at
   `.sdlc/artifacts/ultra-review.md`. Preserve the finding structure
   (severity, file, location, explanation).

3. Triage every finding:
   - **Fix:** Apply the code change now.
   - **Defer:** Out of scope; open a follow-up ticket.
   - **Reject:** Not a real issue; record rationale in one sentence.

4. Append triage decisions under a `## Actions` section.
5. Apply accepted fixes in a single commit:
   `fix(<scope>): address ultra-review findings`
6. Re-run the test suite (Step 6) to confirm no regressions.

4. PIPESTATUS index made explicit (execute.sh) correctness
orchestrator/lib/execute.sh tests/execute_integration_test.sh

Objective

Make sure the runner returns the Claude CLI's exit code — not the always-zero printf that feeds its stdin — so the retry loop in run-pipeline.sh sees real failures.

Why this change exists

The pipeline is printf ... | sdlc_run_with_timeout claude ... | tee. PIPESTATUS[0] is printf (always 0), PIPESTATUS[1] is claude, PIPESTATUS[2] is tee. The pre-existing Codex code correctly used ${pipe_status[1]}. During the runner rewrite, an intermediate draft (caught in Step 05 testing) used PIPESTATUS[0], which would have silently hidden every Claude failure and timeout. The committed code now uses PIPESTATUS[1] with an inline comment pinning the invariant.

Justification audit

  • Strong Inline comment "claude is PIPESTATUS[1]; PIPESTATUS[0] is always printf's success" captures the invariant so a future pipeline reshuffle can't silently regress it.
  • Strong The subshell (cd $repo_root; ... ; exit "${PIPESTATUS[1]}") decouples PIPESTATUS handling from the outer retry loop, making local exit_code=$? unambiguously the claude exit code.
  • Strong New integration test "run_claude_step propagates claude exit code" in execute_integration_test.sh stubs a non-zero claude and asserts the function returns non-zero. Pins the invariant against accidental regressions of PIPESTATUS index.
(

  printf '%s' "$full_prompt" | sdlc_run_with_timeout "$timeout_seconds" \
    claude \
      --print \
      --model "$CLAUDE_MODEL" \
      --effort "$CLAUDE_EFFORT" \
      --permission-mode "$permission_mode" \
      --output-format "$CLAUDE_OUTPUT_FORMAT" \
      ${claude_args[@]+"${claude_args[@]}"} \
      2>&1 | tee "$log_file"
  # Pipeline is: printf | sdlc_run_with_timeout claude ... | tee. The claude
  # (and timeout-wrapper) exit code is PIPESTATUS[1]; PIPESTATUS[0] is always
  # printf's success and would mask real failures from the retry loop.
  exit "${PIPESTATUS[1]}"
