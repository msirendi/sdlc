SDLC-3: CLI UX — help flag, in-run feedback, Ctrl+C handling

This branch fixes three operator-facing ergonomics gaps in the sdlc CLI. A new --help/--version surface makes the tool self-documenting without booting the orchestrator; a heartbeat loop plus a tail -f hint keep the terminal alive during long Claude steps; and a SIGINT trap that walks the Claude subshell's process tree finally makes Ctrl+C actually stop the pipeline. The most load-bearing change is the subtle one in execute.sh: backgrounding the Claude subshell with & so that bash can deliver signals to the orchestrator while wait blocks — without that, all the trap machinery would never fire.

Block 1: First-class --help / --version surface in sdlc

bin/sdlc

Objective

Give a new operator a way to discover what sdlc does — typical flow, step map, artifact layout, and Ctrl+C behavior — without needing to read source files and without triggering the orchestrator. Also expose the install location and revision so operators can confirm which checkout is on PATH.

Why this change exists

The previous binary had no --help — invoking sdlc with any argument fell straight into the orchestrator's internal option parser, which spoke to the machine and not to humans. The task explicitly names this as one of three operator-facing gaps: "there is no first-class help screen." Help must also work outside a git repo, because a brand-new operator hasn't yet bootstrapped a target — so the intercept has to happen before the orchestrator does any repo-rooted work.

Justification audit

  • Strongly justified
    print_help() with USAGE, TYPICAL FLOW, PIPELINE STEPS, DURING A RUN, and a cross-reference to sdlc-status. The acceptance criteria enumerate each of these sections, and the unit tests assert on their presence.
  • Strongly justified
    print_version() printing sdlc (SDLC_HOME=…, revision=…). Explicit AC; the revision lookup is guarded so it degrades to unknown on non-git installs rather than erroring.
  • Strongly justified
    Argument intercept loop before repo_path parsing. This is the non-obvious piece: it guarantees --help works outside a git repo (covered by test_sdlc_help_short_circuits_orchestrator_without_git_repo) and that any position of --help in argv works.
  • Probably justified
    The one-line header comment addition (sdlc --help # show user-facing help). Mechanical doc churn — keeps the usage banner in sync with the new flag but not required by any AC.

Behavioral impact

  • Two new exit-0 code paths for sdlc --help, sdlc -h, and sdlc --version.
  • These short-circuit before repo_path validation, so they no longer require a git repo under $PWD.
  • No change to existing flag parsing (--dry-run, --start-from, --only, --include-manual).

Reviewer scrutiny

  • The intercept matches any arg named -h/--help/--version anywhere in argv. Confirm this doesn't collide with a future flag that takes those strings as a value (currently none do — all flags that take values use =-style parsing in run-pipeline.sh).
  • $SDLC_HOME in the help text appears inside a <<'EOF' heredoc — it is printed literally, not expanded. Confirmed intentional: the user sees the variable name in the SEE ALSO line as a pointer. print_version(), by contrast, uses printf with interpolation.
  • Revision lookup guards on [[ -d "$SDLC_HOME/.git" ]] — this gracefully handles git worktree checkouts where .git is a file, falling back to unknown. Confirm this is acceptable for worktree installs (operator would see revision=unknown).

print_version() {

Block 2: Signal-interruptible step execution (Ctrl+C actually works)

orchestrator/lib/execute.sh orchestrator/run-pipeline.sh

Objective

When the operator presses Ctrl+C during a long Claude step, the orchestrator must terminate the step and every descendant process it spawned (the tee helpers, the timeout wrapper, the Claude CLI, and any children Claude may have spawned), then exit with the conventional SIGINT code 130 — leaving no orphan claude or sleep processes behind.

Why this change exists

Before this change, Ctrl+C was practically useless: the orchestrator ran the Claude step in a synchronous subshell (…) inside a pipeline. In bash, signal delivery to the parent is deferred until the foreground command returns, so the trap (even if one had been installed) would not fire until Claude finished on its own. The fix has three linked pieces: (a) background the subshell with & and wait (wait is signal-interruptible); (b) write a trap that walks the process tree because the subshell's descendants inherit the orchestrator's pgid and kill -PGID would also kill the orchestrator itself; (c) quiet bash's own "Terminated: 15" job-control notice so it doesn't clutter the terminal.

Justification audit

  • Strongly justified
    Backgrounding the Claude subshell () &) + wait "$CURRENT_STEP_PID". This is the linchpin — without it, no trap fires during a step.
  • Strongly justified
    CURRENT_STEP_PID as an unscoped global (intentionally not local, and documented as such in the comment). The interrupt handler reads it from the parent scope.
  • Strongly justified
    signal_process_tree() using pgrep -P recursively. The task explicitly requires macOS-compatible process-tree traversal and rules out GNU coreutils.
  • Strongly justified
    terminate_current_step() with TERM → poll(2s, 100ms slices) → KILL escalation. This is graceful-first, forceful-only-if-needed, which is the right etiquette for a long-running Claude process that may be mid-write.
  • Strongly justified
    handle_interrupt() printing the warning, stopping heartbeat, terminating the step, exiting 130. Every sub-action is a named AC requirement.
  • Probably justified
    exec 2>>"$log_file" inside the backgrounded subshell. Suppresses bash's post-kill "Terminated: 15" chatter — a real ergonomics gain, but with a small cost: any unexpected subshell-level stderr (not Claude's — Claude has its own 2> >(tee…) redirect) now goes only to the log. Reviewers should be comfortable that this swallows no diagnostic the operator currently relies on.
  • Weakly covered by tests
    There is no automated test for the SIGINT contract. The AC states "Under a PTY, sending SIGINT to sdlc during a step exits with code 130, leaves no lingering claude / sleep descendants" — but the test suite only exercises --help, -h, --version, and HEARTBEAT_INTERVAL. Signal-handling tests are admittedly annoying (need a PTY harness and a sacrificial long-running step fixture), but this is the highest-complexity code in the branch and has zero automated regression coverage.

Behavioral impact

  • Exit-code propagation: wait "$CURRENT_STEP_PID" returns the subshell's exit code, which is then captured into exit_code exactly as the retry loop previously expected. Timeout (124), validation failure (2), and success (0) all still flow through unchanged.
  • On SIGINT: orchestrator prints Interrupt received. Terminating current step and exiting..., followed by Pipeline halted by user (SIGINT)., then exits 130.
  • On SIGTERM (same trap): identical behavior.
  • Subshell's own stderr (bash job-control notices, any accidental internal errors) now goes to the per-step log instead of the terminal.

Reviewer scrutiny

  • Confirm wait $PID reliably returns the subshell's exit status even after the trap fires — it does, because the handler calls wait "$pid" 2>/dev/null || true and then exit 130, so the parent never returns to the retry loop's exit_code=$? line after an interrupt. No ambiguity.
  • signal_process_tree is recursive and unbounded. A pathological Claude child fork bomb could blow the bash call stack. Acceptable risk given the use case.
  • The TERM→KILL poll uses sleep 0.1 × 20 = 2s. Is 2s enough for Claude's stdout tee to flush? If not, a KILL could lose trailing summary content. Consider whether this window is tunable.
  • The exec 2>>"$log_file" happens inside the subshell, so Claude's own stderr redirect (2> >(tee …)) later in the pipeline wins for Claude — good. But confirm this is the intended layering: subshell-meta stderr → log; Claude stderr → tee → terminal + log.

Signal-handling scaffolding
# Globals the interrupt handler reads. execute.sh sets CURRENT_STEP_PID to the
# PID of the backgrounded Claude subshell; the heartbeat loop below sets
# STEP_HEARTBEAT_PID. Both stay empty when no step is active.
CURRENT_STEP_PID=""
STEP_HEARTBEAT_PID=""

signal_process_tree() {
  local signal="$1"
  local root="$2"
  local child
  for child in $(pgrep -P "$root" 2>/dev/null || true); do
    signal_process_tree "$signal" "$child"
  done
  kill "-$signal" "$root" 2>/dev/null || true
}

terminate_current_step() {
  local pid="$CURRENT_STEP_PID"
  [[ -z "$pid" ]] && return 0
  if kill -0 "$pid" 2>/dev/null; then
    signal_process_tree TERM "$pid"
    local waited=0
    while [[ "$waited" -lt 20 ]] && kill -0 "$pid" 2>/dev/null; do
      sleep 0.1
      waited=$((waited + 1))
    done
    if kill -0 "$pid" 2>/dev/null; then
      signal_process_tree KILL "$pid"
    fi
  fi
  wait "$pid" 2>/dev/null || true
  CURRENT_STEP_PID=""
}

handle_interrupt() {
  printf '\n' >&2
  sdlc_log "WARN" "Interrupt received. Terminating current step and exiting..."
  stop_heartbeat
  terminate_current_step
  sdlc_log "WARN" "Pipeline halted by user (SIGINT)."
  # 130 = 128 + SIGINT(2), the conventional shell exit code for Ctrl+C.
  exit 130
}

trap handle_interrupt INT TERM

Block 3: Heartbeat loop so long steps aren't silent

orchestrator/config.sh orchestrator/run-pipeline.sh tests/config_unit_test.sh

Objective

Emit a periodic <step> still running (elapsed: Ns) line while a Claude step is in flight, so the terminal never goes silent for minutes at a time. Make the cadence tunable and disable-able via a new HEARTBEAT_INTERVAL knob.

Why this change exists

Claude steps routinely take 2–5 minutes. With the old behavior, an operator had no way to tell whether the pipeline was progressing or had silently deadlocked. The task explicitly frames this as an ergonomics gap: "the terminal goes silent for minutes at a time." The heartbeat is a second-choice solution to streaming Claude's actual progress — Claude's --output-format text contract (preserved by technical constraint for downstream steps 9/10) gives no intermediate stream, so an out-of-band timer is the cheapest workable signal.

Justification audit

  • Strongly justified
    HEARTBEAT_INTERVAL="${HEARTBEAT_INTERVAL:-120}" in config.sh. Default 120s is explicit in the AC; environment-override path is standard.
  • Strongly justified
    emit_heartbeat_loop() using sleep & wait (rather than raw sleep). Raw sleep blocks signal delivery in a bash subshell the same way a synchronous pipeline does — the author clearly hit this and documented it.
  • Strongly justified
    Per-loop trap 'kill "$sleep_pid" 2>/dev/null; exit 0' TERM INT. This is what stop_heartbeat relies on in Block 2 — a tight handoff between the signal path and the heartbeat subshell.
  • Strongly justified
    Conditional spawn in execute_step_with_retries: only start when HEARTBEAT_INTERVAL > 0. Honors the "0 disables heartbeats" AC.
  • Strongly justified
    stop_heartbeat called in both the happy path (after wait) and the interrupt path (from handle_interrupt). Double-cleanup is idempotent because the function no-ops on empty/dead PID.
  • Weakly covered by tests
    Tests assert the config contract (default 120, env override) but do not assert that the loop actually fires, emits the expected log line, or stops cleanly. A 2-second cadence + a no-op fixture step would catch regressions like a broken sleep & wait or a typo in the log format. Consider adding one.

Behavioral impact

  • During each step, a child process prints an INFO line every HEARTBEAT_INTERVAL seconds.
  • Log format: <label> (attempt N) still running (elapsed: Ms).
  • Setting HEARTBEAT_INTERVAL=0 disables the feature entirely (no heartbeat subshell is spawned).
  • Elapsed time is measured via $SECONDS, which is per-subshell and resets correctly on each attempt.

Reviewer scrutiny

  • $SECONDS is integer-only, so elapsed time rounds down at the second boundary. Acceptable.
  • The [[ "$HEARTBEAT_INTERVAL" -gt 0 ]] check will error if HEARTBEAT_INTERVAL is set to a non-numeric value (e.g. "off"). The docs only claim numeric support, so this is probably fine — but an operator who tries HEARTBEAT_INTERVAL="" via an empty override will crash the pipeline. Consider clamping.
  • Heartbeat lines land in the orchestrator log alongside Claude's streamed stdout (via tee) — confirm they don't interleave confusingly in the per-step log.

stop_heartbeat() {
  if [[ -n "$STEP_HEARTBEAT_PID" ]] && kill -0 "$STEP_HEARTBEAT_PID" 2>/dev/null; then
    kill -TERM "$STEP_HEARTBEAT_PID" 2>/dev/null || true
    wait "$STEP_HEARTBEAT_PID" 2>/dev/null || true
  fi
  STEP_HEARTBEAT_PID=""
}

emit_heartbeat_loop() {
  local label="$1"
  local interval="$2"
  local start=$SECONDS
  local sleep_pid=""
  # `wait` (unlike the raw `sleep` builtin) is interruptible by signals
  # delivered to this subshell. The TERM trap kills the sleep grandchild
  # so it doesn't linger, then exits cleanly.
  trap 'kill "$sleep_pid" 2>/dev/null; exit 0' TERM INT
  while :; do
    sleep "$interval" &
    sleep_pid=$!
    wait "$sleep_pid" 2>/dev/null || true
    sleep_pid=""
    local elapsed=$((SECONDS - start))
    sdlc_log "INFO" "$label still running (elapsed: ${elapsed}s)"
  done
}

Block 4: In-run feedback — tail -f hint on start, Status excerpt on success

orchestrator/lib/execute.sh orchestrator/run-pipeline.sh

Objective

Two small signal-boosts around each step: at start, tell the operator how to tail the log from another terminal; at success, echo the one-line Status field from the step's final summary back to the terminal so they don't have to open the summary file to see what happened.

Why this change exists

These round out the "in-run feedback" half of the ergonomics fix. The tail hint is an AC item ("must print a Follow progress: tail -f <log> hint"). The Status excerpt is the terminal counterpart to the heartbeat: the heartbeat tells you a step is alive, the excerpt tells you what it accomplished. Together they close the feedback loop without forcing the operator to read the summary file.

Justification audit

  • Strongly justified
    sdlc_log "INFO" "Follow progress: tail -f $log_file" added to the step-launch banner. Literal AC requirement.
  • Strongly justified
    emit_step_summary_excerpt() called only on success (after update_context). Explicit AC: "When a step completes successfully, the orchestrator must print a one-line excerpt from the step summary."
  • Probably justified
    The regex ^[[:space:]]*(5\.[[:space:]]*)?Status: accepts both raw Status: and the numbered form 5. Status: used by the final-response template. Future-proof against trivial format drift, but tightly coupled to the current step-file convention that Status is field #5.
  • Probably justified
    Early-return on empty summary ([[ ! -s "$summary_path" ]] && return 0). Silences a known edge case (timeout with no summary), consistent with the rest of the orchestrator's "degrade quietly" policy.

Behavioral impact

  • Orchestrator log (and terminal) gains two new INFO lines per step: Follow progress: … at start, <label>: Status: READY|BLOCKED|… at success.
  • No impact on the step contract itself (summary files, exit codes, validation).
  • No excerpt is emitted on timeout or validation failure — the existing WARN lines cover those paths.

Reviewer scrutiny

  • The regex hard-codes the numeric prefix 5.. If a step file ever reorders its final-response sections (task definitions currently all use the same template, but that's a convention not a contract), the excerpt silently goes missing. Consider documenting the contract in CLAUDE.md or a step-template file.
  • grep -E … | tail -n 1 picks the last match — good for a multi-pass summary, but if a step's prose contains the word "Status:" earlier (e.g. "Status: this section will explain..."), the excerpt could pick up the wrong line. Unlikely in practice.
  • No test asserts the excerpt is emitted. Minor — the output is informational, not contractual.

# Print a short, terminal-friendly excerpt of the step's final response so
# the operator sees what the step accomplished without hunting through
# logs. Looks for the canonical Status line.
emit_step_summary_excerpt() {
  local label="$1"
  local summary_path="$2"
  [[ ! -s "$summary_path" ]] && return 0

  local status_line
  status_line=$(grep -E '^[[:space:]]*(5\.[[:space:]]*)?Status:' "$summary_path" | tail -n 1 || true)
  if [[ -n "$status_line" ]]; then
    sdlc_log "INFO" "$label: $(printf '%s' "$status_line" | sed -E 's/^[[:space:]]*//')"
  fi
}
