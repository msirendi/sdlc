Block 1: Implement a repo-aware latest-run summarizer

orchestrator/status.sh

Objective

Expose the last pipeline run as a concise, human-readable status summary without forcing operators to inspect .sdlc/logs/ manually.

Why This Change Exists

The ticket’s core requirement is operational visibility: the pipeline already emits a manifest and orchestrator log, but there was no lightweight reader for them. This block turns those artifacts into a stable operator command while staying inside the repo’s Bash-only, no-extra-runtime constraints.

Justification Audit

Strongly justified Sourcing config.sh and lib/common.sh, reusing sdlc_log, and resolving the repository with git rev-parse --show-toplevel all directly support the stated requirement to run from any directory inside the target repo while matching existing orchestrator conventions.
Strongly justified find_latest_run_dir selects the lexically newest child directory under .sdlc/logs/, which matches the pipeline’s timestamp-based run IDs and avoids any need for external tools.
Strongly justified parse_manifest uses the manifest as the source of truth for planned-step order and repository path. That is the correct artifact to preserve the intended step list, especially when some planned steps never completed.
Probably justified parse_orchestrator_log scrapes exact human-readable phrases such as Repository:, Completed, Pipeline halted at, and Run complete. This is valid against today’s producer, but it creates a brittle coupling: wording changes in run-pipeline.sh will break status parsing even if the pipeline still works.
Strongly justified The summary renderer maps planned steps to ✓ completed, ✗ failed, and – skipped, humanizes elapsed seconds, and prints the log directory. Those are exactly the user-facing fields requested by the ticket.
Probably justified Treating a malformed newest run as a hard error is defensible because it avoids silently reporting stale information from an older run. It is still a policy choice rather than an explicit requirement, so the reviewer should confirm that “latest run is corrupt” is intended to be fatal.
Probably justified The main guard is not operator-visible, but it is coherent with the later test additions because it allows the helper functions to be sourced safely without changing CLI behavior.

Behavioral Impact

  • Introduces a new command path: bash orchestrator/status.sh.
  • Works from nested directories inside a governed repository because it re-roots via Git.
  • Reports no runs as a successful, non-error condition.
  • Marks incomplete or interrupted runs by leaving elapsed time unavailable and leaving remaining planned steps skipped.
  • Fails fast if the newest run directory is missing required artifacts.

Reviewer Scrutiny

  • Confirm that run-pipeline.sh log text is a stable contract, or accept that status.sh must be updated alongside any wording change.
  • Verify whether dry-run output should be treated as “unavailable / interrupted” or deserves a dedicated dry-run status instead.
  • Decide whether the command should fall back to the newest valid run when the newest directory is corrupt.
  • Verify that .sdlc/logs/ will only contain sortable run directories and no stray folders that could sort after a timestamp.

orchestrator/status.sh
repo_root=$(git rev-parse --show-toplevel)
logs_dir="$repo_root/$LOGS_DIR_REL"

if ! latest_run_dir=$(find_latest_run_dir "$logs_dir"); then
  printf 'No pipeline runs found.\n'
  exit 0
fi

manifest_file="$latest_run_dir/pipeline-manifest.md"
orchestrator_log="$latest_run_dir/orchestrator.log"

parse_manifest "$manifest_file"
parse_orchestrator_log "$orchestrator_log"

for step_name in "${PLANNED_STEPS[@]}"; do
  if [[ -n "$FAILED_STEP" && "$step_name" == "$FAILED_STEP" ]]; then
    printf '  ✗ failed %s\n' "$step_name"
    continue
  fi

  if array_contains "$step_name" "${COMPLETED_STEPS[@]}"; then
    printf '  ✓ completed %s\n' "$step_name"
    continue
  fi

  printf '  – skipped %s\n' "$step_name"
done

if [[ -n "$ELAPSED_SECONDS" ]]; then
  printf 'Elapsed: %s\n' "$(format_duration "$ELAPSED_SECONDS")"
else
  printf 'Elapsed: unavailable\n'
  printf 'Note: the latest run may still be in progress or was interrupted.\n'
fi

README.md
## Checking run status

Use the status command from anywhere inside a governed repository to inspect the
latest pipeline run without browsing `.sdlc/logs/` manually:

```bash
bash "$SDLC_HOME/orchestrator/status.sh"
```

The summary reports the latest run ID, repository name, per-step outcome,
elapsed time, and the log directory path. Step states are shown as:

- `✓ completed`: the step finished successfully
- `✗ failed`: the pipeline halted at that step
- `– skipped`: the step was planned for the run but did not complete

```bash
sdlc-status() {
  SDLC_HOME="${SDLC_HOME:-/path/to/sdlc}" \
    bash "$SDLC_HOME/orchestrator/status.sh"
}
```

capture_command() {
  local workdir="$1"
  shift

  set +e
  CAPTURED_OUTPUT=$(cd "$workdir" && "$@" 2>&1)
  CAPTURED_STATUS=$?
  set -e
}

write_manifest() {
  local run_dir="$1"
  local repo_path="$2"
  shift 2

  {
    printf '# Pipeline Run Manifest\n\n'
    printf -- '- Repository: `%s`\n' "$repo_path"
    printf '## Planned steps\n'
    for step_name in "$@"; do
      printf -- '- `%s` (automated, timeout 1800s)\n' "$step_name"
    done
  } > "$run_dir/pipeline-manifest.md"
}

test_parse_orchestrator_log_collects_completed_failed_and_elapsed_data() {
  local log_file="$TEST_TEMP_DIR/orchestrator.log"

  cat <<'EOF' > "$log_file"
[2026-04-13 14:25:31] [INFO] Repository: fixture-repo
[2026-04-13 14:30:54] [INFO] Completed 03-implement.md
[2026-04-13 14:24:14] [ERROR] Pipeline halted at 04-agents-md-check.md
[2026-04-13 14:24:14] [INFO] Run complete: 2 succeeded, 1 failed, 287s elapsed.
EOF

  parse_orchestrator_log "$log_file"

  assert_equals "fixture-repo" "$REPO_NAME"
  assert_equals "04-agents-md-check.md" "$FAILED_STEP"
  assert_equals "287" "$ELAPSED_SECONDS"
}

Overall Review Summary

Overall PR Objective

Add an operator-facing sdlc-status workflow that summarizes the most recent pipeline run from existing artifacts, document how to invoke it, and protect the behavior with native shell tests.

Potentially Superfluous Changes

No production edit looks obviously superfluous. The only arguable stretch is the size of tests/testlib.sh; it is coherent support code, but it establishes a custom shell-test framework that is larger than this feature alone requires.

Changes That Need Author Explanation

  • Is failing on the malformed newest run an intentional user contract, or should the command fall back to the newest valid run?
  • Should dry runs be reported distinctly rather than as “elapsed unavailable / possibly interrupted”?
  • Is lexical directory ordering guaranteed for every future entry under .sdlc/logs/?

Missing or Weak Test Coverage

  • No fixture covers the orchestrator’s dry-run output shape.
  • No test covers stray non-run directories or partially-created newest directories beyond the missing-manifest case.
  • No explicit case covers runs that include manual steps.

Reviewer Questions

  • Is the human-readable orchestrator log intended to be a parsing contract, or should a machine-readable status artifact exist instead?
  • Does the repository want to continue building custom Bash test utilities for future shell features?
  • Should status reporting privilege recency, or validity, when the newest run artifact set is incomplete?

Assessment

The branch stays close to scope and the production changes are easy to justify. Review should focus less on whether the feature belongs and more on the long-term contract choices: parsing stability, malformed latest-run policy, and test-harness direction.
