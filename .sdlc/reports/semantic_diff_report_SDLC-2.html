SDLC-2: Decouple test authoring, execution, and repair

This branch restructures the pipeline around one idea: each Claude invocation should have a single job. Tests are now authored (Step 3) before implementation (Step 4), running tests (Step 6) is a pure reporter, and fixing failing tests (Step 7) is a separate code-only pass. The orchestrator, not the agent, drives the 6↔7 loop. Most of the churn — the 03→17 renumbering of step files and the equivalent pointer updates in tests, README, and overrides — is mechanical. The genuinely novel pieces are the Result: parse contract, the test-fix loop helper, the manifest filter that suppresses duplicate top-level Step 7 runs, and four new PATH-installable bin/ wrappers.

Block 1: Decouple test authoring, implementation, execution, and repair into four numbered steps

03-tests.md 04-implement.md 06-run-tests.md 07-fix-test-failures.md 02-technical-spec.md

Objective

Split what used to be a single implement-and-test loop into four purpose-isolated step files so that each Claude Code invocation has exactly one responsibility: write the contract, implement against it, run it, or fix the code that violates it.

Why this change exists

The staged requirements explicitly call out the problem: under the old ordering Step 3 was implementation and Step 5 was tests, which let implementations shape tests rather than the other way around, and the old Step 6 combined running tests with fixing them, conflating diagnosis with repair. The new files are pure contracts — 03 only writes tests, 04 only implements against committed tests, 06 only runs, 07 only fixes. Each guardrail section explicitly forbids the other steps' responsibilities.

Justification audit

Strongly justified 03-tests.md derives tests from .sdlc/artifacts/technical-spec.md only, commits them in a red state, and returns BLOCKED if the spec is ambiguous. This is the mechanism by which the spec becomes an executable contract for Step 4 — a direct encoding of the acceptance criterion.
Strongly justified 04-implement.md explicitly forbids running tests and editing test files; it routes BLOCKED back to Step 2 (spec) or Step 3 (tests) when the contract is wrong. This is what keeps the implementer from silently rewriting the tests to match convenient code.
Strongly justified 06-run-tests.md mandates the first non-blank Result: PASS|FAIL line and spells out the exact report schema the orchestrator parses. The guardrail "Do not write Result: PASS to make the pipeline proceed" directly defends the loop contract.
Strongly justified 07-fix-test-failures.md reads the report, treats tests as read-only, forbids test-suite execution, and exits BLOCKED when a test is genuinely wrong. Every rule here exists to keep the 6↔7 loop's iteration count honest.
Probably justified 02-technical-spec.md gains a paragraph that tells the spec author to be concrete enough that Step 3 can derive test cases without inventing details. This is an upstream nudge rather than an enforcement — the guardrail that actually protects the contract lives in Step 3's BLOCKED rule.
Probably justified The cross-references between the step files ("Step 6 owns execution", "Step 7 owns repair", etc.) are load-bearing for the reader, but they do drift risk into the four files — if one is renamed or resequenced in a future ticket, all four must be updated together. This is accepted cost, not a defect.
Probably justified The renumbering cascade (05-agents-md-check through 17-cleanup) is mechanical. It is justified because the new tests-before-implementation ordering forces it, but it is also the bulk of the diff, and each renamed file is an opportunity for an orphaned reference elsewhere. The accompanying test updates (Block 6) are how we know nothing was missed.

Behavioral impact

  • Running sdlc --only 03-tests.md now expects to produce red tests and exit READY, not a green run.
  • Step 4 will refuse to run the suite; any prior habit of npm test at the end of implementation is explicitly forbidden.
  • Step 6 no longer modifies code: a red suite means Step 6 exits cleanly with Result: FAIL, and the orchestrator (not the agent) schedules Step 7.
  • Step 7 never edits test files — a genuinely wrong test routes back to Step 2/3 via BLOCKED.

Block 2: Orchestrator drives the 06↔07 test-fix loop

orchestrator/run-pipeline.sh orchestrator/lib/test_fix_loop.sh

Objective

Make the orchestrator — not the Claude agent — the authority that decides when to re-run tests and when to invoke the fix pass. Step 6 writes a parseable Result: marker; execute_test_loop() reads it and iterates Step 7 → Step 6 until the report says PASS or the cap is hit.

Why this change exists

With the old combined run-and-fix step, the agent owned its own retry behavior and the orchestrator had no visibility into iteration count. That made bounded failure impossible: a never-passing suite could loop indefinitely inside one step's timeout, and the pipeline had no stable artifact to inspect between attempts. Extracting the loop into Bash gives the orchestrator a hard iteration cap, per-iteration log separation, and a single source of truth (.sdlc/artifacts/test-results.md) for the pass/fail decision.

Justification audit

Strongly justified sdlc_test_results_status() isolates the parse contract in one place: lower-case and leading-whitespace tolerant, first-line-wins, UNKNOWN on missing/malformed input. The comment "treats UNKNOWN as a non-pass ... silently skipping [fix] after a malformed report would let red branches sail through to the open-PR step" captures the security-over-convenience policy.
Strongly justified execute_step_with_retries() is extracted from the old inline loop and gains an iter_suffix parameter so Step 6 and Step 7 can be re-invoked without their _attempt*.log files overwriting each other. The prior inline form could not have been re-entered safely.
Strongly justified execute_test_loop() always runs Step 6 first (so the report exists), short-circuits on PASS, and otherwise iterates up to MAX_TEST_FIX_ITERATIONS (default 3). The iteration cap makes stuck fix loops fail fast instead of consuming the whole pipeline timeout.
Strongly justified The manifest filter ("drop Step 07 from FILTERED_STEPS when Step 06 is planned") prevents the outer for loop from running Step 7 a second time after the inner loop already ran it. Crucially, it preserves the escape hatch where --only 07-fix-test-failures.md without Step 6 keeps Step 7 in the plan.
Probably justified Treating the fix step's step-runner failure (exit of execute_step_with_retries) as a hard halt for the whole pipeline is the right default, but it means a single flaky fix attempt can surface as a pipeline failure rather than retrying at the loop layer. Reviewers should confirm the retry budget on 07-fix-test-failures.md (currently 2) is enough before the loop gives up.
Probably justified The inner-loop log suffix scheme (_iter1, _iter2) is a reasonable scheme but the status command and any log-reading tooling must be aware of it; the summary in sdlc-status may flatten iterations into a single line.

Behavioral impact

  • A failing suite now produces a deterministic sequence: 06, 07(iter 1), 06(iter 1), 07(iter 2), 06(iter 2), …
  • A malformed test-results.md (parsed as UNKNOWN) invokes Step 7 and re-runs Step 6 rather than being quietly accepted.
  • Step 7 no longer appears as a standalone line in the planned-steps manifest when Step 6 is also planned.
  • Operators retain direct access: sdlc --only 07-fix-test-failures.md runs Step 7 alone (no loop).

Reviewer scrutiny

  • Confirm the UNKNOWN → fix-runs-anyway policy is desired. The alternative (halt-on-malformed) would be less forgiving but catches a broken Step 6 sooner.
  • Verify execute_step_with_retries's local exit_code=$? capture still respects set -e behavior inside its callers — it is guarded by set +e / set -e, but the extraction widens the surface area.
  • Check that the removal of the original success=false / break pattern did not drop a retry path (every retry attempt must still be visible in the log directory).

#!/usr/bin/env bash

# Read the first `Result:` line from a test-results report and echo PASS,
# FAIL, or UNKNOWN. The orchestrator treats UNKNOWN as a non-pass: the fix
# step still runs, because silently skipping it after a malformed report
# would let red branches sail through to the open-PR step.
sdlc_test_results_status() {
  local results_file="$1"
  if [[ -z "$results_file" || ! -f "$results_file" ]]; then
    printf 'UNKNOWN\n'
    return
  fi
  local marker
  marker=$(sed -n 's/^[[:space:]]*Result:[[:space:]]*//Ip' "$results_file" | head -n 1)
  marker=$(printf '%s' "$marker" | tr '[:lower:]' '[:upper:]')
  case "$marker" in
    PASS*) printf 'PASS\n' ;;
    FAIL*) printf 'FAIL\n' ;;
    *)     printf 'UNKNOWN\n' ;;
  esac
}

orchestrator/run-pipeline.sh · L299–L345
# Drive the run-tests / fix-test-failures loop. Step 6 always runs first so
# the report exists. After each Step 6 result, if it is FAIL/UNKNOWN we run
# Step 7 and re-run Step 6, up to MAX_TEST_FIX_ITERATIONS fix passes.
execute_test_loop() {
  local run_step_file="$1"
  local fix_step_file="$STEPS_DIR/$TEST_FIX_STEP"
  local results_file="$REPO_ROOT/$TEST_RESULTS_REL"
  local results_status

  if [[ ! -f "$fix_step_file" ]]; then
    sdlc_log "ERROR" "Fix step file is missing: $fix_step_file"
    return 1
  fi

  if ! execute_step_with_retries "$run_step_file"; then
    return 1
  fi

  results_status=$(sdlc_test_results_status "$results_file")
  if [[ "$results_status" == "PASS" ]]; then
    sdlc_log "INFO" "Test suite is green after the initial run; skipping $TEST_FIX_STEP."
    return 0
  fi

  local iter=1
  while [[ "$iter" -le "$MAX_TEST_FIX_ITERATIONS" ]]; do
    sdlc_log "INFO" "Test results: $results_status. Running fix iteration $iter/$MAX_TEST_FIX_ITERATIONS."
    if ! execute_step_with_retries "$fix_step_file" "$iter"; then
      sdlc_log "ERROR" "Fix step $TEST_FIX_STEP failed on iteration $iter."
      return 1
    fi

    if ! execute_step_with_retries "$run_step_file" "$iter"; then
      sdlc_log "ERROR" "Re-run of $TEST_RUN_STEP failed on iteration $iter."
      return 1
    fi

    results_status=$(sdlc_test_results_status "$results_file")
    if [[ "$results_status" == "PASS" ]]; then
      sdlc_log "INFO" "Test suite passed after fix iteration $iter."
      return 0
    fi

    iter=$((iter + 1))
  done

  sdlc_log "ERROR" \
    "Test suite still $results_status after $MAX_TEST_FIX_ITERATIONS fix iterations; halting."
  return 1
}

Block 4: PATH-installable command wrappers under bin/

bin/sdlc bin/sdlc-dry bin/sdlc-init bin/sdlc-status orchestrator/init-target-repo.sh

Objective

Ship four real executable wrappers so operators can run the pipeline from any directory without first exporting SDLC_HOME or knowing internal script paths.

Why this change exists

Previously the README prescribed shell-function shims that each repo had to paste into ~/.zshrc. That coupled the repo's usability to the operator's dotfiles and meant a freshly-cloned checkout had no direct entry point. Real scripts in bin/ (a convention npm, pnpm, uv, etc. all lean on) let operators prepend one directory to PATH once and have four commands immediately.

Justification audit

Strongly justified The symlink-safe _resolve_script_dir helper is what makes ln -s ~/sdlc/bin/sdlc /usr/local/bin/sdlc work — without it, SDLC_HOME would resolve to /usr/local. Every wrapper duplicates the helper for isolation.
Strongly justified bin/sdlc peels the first positional arg as an optional repo_path only when it is not a --flag, so sdlc --dry-run and sdlc /path/to/repo --dry-run both work.
Strongly justified bin/sdlc-dry is a thin exec $_BIN_DIR/sdlc "$@" --dry-run — no logic duplication, and the flag is appended so the caller cannot accidentally override it.
Strongly justified bin/sdlc-status exists at all. The prior README only suggested a shell function; now which sdlc-status actually resolves, which is what the new tests pin.
Probably justified init-target-repo.sh's instructional footer switched from a full-path fallback to export PATH="$SDLC_HOME/bin:$PATH". The guidance is cleaner, but a user who already set SDLC_HOME without touching PATH will now see a reference to sdlc-dry they cannot run. The fallback line mitigates this, but it's worth confirming the wording is clear.

Behavioral impact

  • After export PATH="$SDLC_HOME/bin:$PATH", sdlc, sdlc-init, sdlc-dry, and sdlc-status are all discoverable globally.
  • Symlinks into /usr/local/bin work without breaking SDLC_HOME.
  • The init script's "Next" banner now points at sdlc-dry rather than orchestrator/run-pipeline.sh --dry-run.

Reviewer scrutiny

  • Verify the while [[ -h "$src" ]]; do … ; done loop terminates on non-symlink endpoints (it does on macOS/Linux via standard readlink, but it's worth a sanity-check under BSD vs GNU).
  • Check that sdlc --help still reaches run-pipeline.sh — the wrapper shifts --help only when it looks like a path, so the flag is forwarded.
  • Confirm sdlc-init's ${1:-$PWD} fallback does not accidentally init $PWD when the operator meant to pass a flag (there are no flags on sdlc-init today, but that is a soft contract).

#!/usr/bin/env bash
# Wrapper that invokes the SDLC pipeline runner from any directory.
set -euo pipefail

_resolve_script_dir() {
  local src="${BASH_SOURCE[0]}"
  while [[ -h "$src" ]]; do
    local dir
    dir="$(cd -P "$(dirname "$src")" >/dev/null 2>&1 && pwd)"
    src="$(readlink "$src")"
    [[ "$src" != /* ]] && src="$dir/$src"
  done
  cd -P "$(dirname "$src")" >/dev/null 2>&1 && pwd
}

_BIN_DIR="$(_resolve_script_dir)"
: "${SDLC_HOME:="$(cd "$_BIN_DIR/.." >/dev/null 2>&1 && pwd)"}"
export SDLC_HOME

repo_path="$PWD"
if [[ $# -gt 0 && "$1" != -* ]]; then
  repo_path="$1"
  shift
fi

exec "$SDLC_HOME/orchestrator/run-pipeline.sh" --repo "$repo_path" "$@"

### Decoupled test workflow

Steps 3, 4, 6, and 7 are intentionally split so each Claude invocation
has one job:

- Step 3 writes the tests from the spec — no implementation, no test
  execution.
- Step 4 implements the code against those tests — no test execution,
  no test edits.
- Step 6 runs the suite and writes .sdlc/artifacts/test-results.md —
  no fixes.
- Step 7 reads the report and fixes production code — no test edits,
  no test execution.

The orchestrator drives a 6↔7 loop: after Step 6 writes its report, if
the first `Result:` line is not `PASS`, Step 7 runs once and Step 6
re-runs. The loop iterates up to MAX_TEST_FIX_ITERATIONS (default 3)
before halting.

test_sdlc_is_discoverable_on_path() {
  # Simulate the documented install step (export PATH="$SDLC_HOME/bin:$PATH")
  # in an isolated environment and confirm each command resolves.
  use_temp_dir
  local fake_home="$TEST_TEMP_DIR/home"
  mkdir -p "$fake_home"

  local resolved
  local name
  for name in sdlc sdlc-init sdlc-dry sdlc-status; do
    resolved=$(PATH="$BIN_DIR:/usr/bin:/bin" HOME="$fake_home" \
      env -i PATH="$BIN_DIR:/usr/bin:/bin" HOME="$fake_home" \
      command -v "$name" || true)
    if [[ -z "$resolved" ]]; then
      fail "$name is not discoverable on PATH when $BIN_DIR is prepended."
    fi
    assert_equals "$BIN_DIR/$name" "$(canonicalize_path "$resolved")" \
      "$name should resolve to the shipped wrapper."
  done
}
